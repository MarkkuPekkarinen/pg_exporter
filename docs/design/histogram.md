# pg_exporter Snapshot Histogram Specification

| Field | Value |
| --- | --- |
| Status | Final |
| Version | 1.2 |
| Last updated | 2026-07-11 |
| Scope | SQL observations aggregated into classic Prometheus bucket series |
| Authority | Normative design for the first pg_exporter Histogram implementation |

## 1. Purpose

This document specifies the first Histogram capability for `pg_exporter`.
It is the source of truth for configuration, SQL result interpretation,
in-memory aggregation, Prometheus exposition, reference collectors, and test
acceptance criteria.

The design intentionally provides one user-facing `HISTOGRAM` usage. Version 1
does not expose separate `GAUGE_HISTOGRAM` and `COUNTER_HISTOGRAM` usages.

The key words **MUST**, **MUST NOT**, **SHOULD**, **SHOULD NOT**, and **MAY**
are to be interpreted as normative requirements.

## 2. Decision summary

Version 1 makes the following decisions:

1. `HISTOGRAM` means a **snapshot distribution rebuilt from SQL rows during
   one real query execution**.
2. Each non-NULL SQL value is one observation. Rows sharing the same complete
   label tuple belong to the same logical Histogram.
3. Histogram state MUST NOT survive a real query execution. A scrape cache hit
   reuses the cached snapshot and MUST NOT add observations.
4. The exporter MUST expose the logical Histogram as cumulative bucket, count,
   and sum **Gauge series**. It MUST NOT use `prometheus.NewConstHistogram` in
   version 1.
5. Bucket series may increase or decrease between snapshots. PromQL consumers
   MUST NOT apply `rate()`, `irate()`, `increase()`, or reset logic to them.
6. Version 1 supports raw observations only. SQL-provided pre-aggregated
   buckets, counter Histograms, formal OpenMetrics GaugeHistogram exposition,
   Prometheus Native Histograms, and OTLP Histograms are out of scope.
7. A query execution that contains Histogram columns is published atomically.
   No scalar or Histogram component from a failed execution may be emitted.

This is a deliberate semantic contract, not an assertion that counter and
gauge Histograms are generally interchangeable. All initial PostgreSQL use
cases describe the current population and therefore have gauge-like temporal
behavior.

## 3. Rationale

The version 1.2 reference use cases are the current per-session distributions
of:

1. active query age;
2. transaction age;
3. client session age;
4. the oldest backend XID horizon age derived from `backend_xid` and
   `backend_xmin`.

Every population can shrink, disappear, or move between buckets. For example,
an active query disappears when it completes, while a running query moves from
smaller age buckets to larger age buckets. These values are not
monotonic across scrapes.

The Go `prometheus.Collector` interface does not require separate collector
implementations for counter and gauge Histograms. The row grouping and bucket
aggregation algorithm is identical. The distinction matters at the temporal
and exposition layers.

`client_golang` v1.23.2 has no high-level constant GaugeHistogram constructor.
`prometheus.NewConstHistogram` produces a counter-like Histogram family. It can
render a correct instant distribution, but it assigns misleading counter
semantics to a snapshot that may decrease. Version 1 therefore emits explicit
Gauge series and accepts that the metric metadata is not a formal
GaugeHistogram family.

## 4. Terminology

**Observation**
: One finite numeric value returned by SQL for a configured Histogram column.

**Label tuple**
: The ordered values of every `LABEL` column in the query, after existing label
  renaming and casting rules are applied.

**Logical Histogram**
: The distribution for one Histogram column and one label tuple during one
  real SQL execution.

**Finite bucket**
: A configured inclusive upper boundary. The `+Inf` bucket is generated by the
  exporter and is never configured.

**Snapshot**
: The complete logical Histogram result produced by one real SQL execution.
  A cached response is the same snapshot, not a new observation period.

## 5. Configuration contract

### 5.1 Column definition

The canonical version 1 syntax reuses the existing `Column.Bucket` field and
its singular YAML key:

```yaml
- query_age_seconds:
    usage: HISTOGRAM
    bucket: [0.01, 0.03, 0.1, 0.3, 1, 3, 10, 30, 100, 300, 1000, 3000, 10000, 30000, 100000, 300000, 1000000, 3000000]
    description: Current active query age snapshot in seconds
```

`usage` remains case-insensitive during parsing and MUST be normalized to
`HISTOGRAM`.

The existing options have these meanings:

- `name` and `rename` determine the logical base metric name in the same way as
  Gauge and Counter columns.
- `scale` is applied to the SQL value before bucket assignment and summation.
- `default`, when explicitly configured, converts SQL NULL to one observation
  with the configured default value. The default is multiplied by `scale`,
  matching the existing scalar metric behavior. Without `default`, NULL is
  ignored.
- `description` is the common help-text prefix for derived series.

### 5.2 Bucket validation

A Histogram column MUST configure at least one finite bucket.

Configured buckets MUST:

- be valid finite `float64` values;
- be strictly increasing;
- contain no duplicates;
- contain neither `NaN` nor positive or negative infinity;
- omit `+Inf`, which is generated automatically;
- use the final exported unit after `scale` has been applied to observations.

Invalid Histogram configuration MUST fail configuration parsing. It MUST NOT
be deferred until scrape time.

Only finite upper boundaries are configured. The first classic bucket has the
implicit mathematical interval `(-Inf, b0]`, where `b0` is the first configured
boundary. Users MUST NOT configure `-Inf`; for finite observations an
`le="-Inf"` bucket would always be empty and carries no information. Values
below `b0`, including negative values, are counted in the first bucket.

For a domain that legitimately contains signed observations, users SHOULD add
finite negative boundaries appropriate to the domain and a `0` boundary, for
example `[-100, -10, -1, 0, 1, 10, 100]`. This preserves useful resolution on
both sides of zero; it does not change the implicit lower edge. The version 1.2
PostgreSQL reference observations are non-negative and are clamped in SQL.

### 5.3 Query column classification

`HISTOGRAM` columns are metric columns. A query is valid when it defines at
least one `GAUGE`, `COUNTER`, or `HISTOGRAM` column.

The existing query-level label model remains unchanged: every logical metric
in one query uses the same ordered label names. If two Histograms require
different labels, they MUST be placed in different query definitions.

### 5.4 Reserved and derived names

For a logical base name `<base>`, version 1 derives:

- `<base>_bucket{le="..."}`;
- `<base>_count`;
- `<base>_sum`.

Within one query definition that contains a Histogram:

- `le` MUST NOT be a query label after label rename;
- derived names MUST be valid Prometheus metric names;
- derived names MUST NOT collide with any scalar or derived metric family in
  that query after rename;
- two Histogram columns MUST NOT resolve to the same base name after rename;
- help text for all instances of each derived family MUST be identical.

Name and query-label conflicts MUST be rejected during per-query configuration
validation. Identical derived families MAY appear in mutually exclusive
version branches when type, help text, label names, and bucket boundaries are
identical.

A runtime constant label named `le` conflicts when any parsed query contains a
Histogram. This MUST be rejected during exporter creation and
configuration reload by the validation path that has both constant labels and
parsed queries. It cannot be validated by `ParseConfig` alone.

## 6. SQL result contract

### 6.1 Raw observations

Version 1 accepts only raw observations. For each result row:

1. label columns form the row's label tuple;
2. each Histogram column independently contributes zero or one observation;
3. a NULL Histogram value contributes no observation unless `default` is set;
4. a finite numeric value contributes exactly one observation after scaling.

Multiple Histogram columns MAY be returned by one SQL query. They are grouped
independently but share the query's label tuple.

### 6.2 Invalid values

An observation that cannot be converted to a finite `float64` is a collection
error. The collector MUST discard the entire new query result, including any
scalar Gauge or Counter metrics already built from earlier rows, and report the
query as failed. It MUST NOT expose a partially updated Histogram snapshot or a
mixture of new scalar metrics and failed Histogram metrics.

All scalar and Histogram component metrics for a query containing Histograms
MUST therefore be buffered outside `q.result` until row iteration, row-error
checking, value validation, and Histogram materialization all succeed. Only
then may the new result be assigned to `q.result`.

The version 1 implementation applies the same pending-result publication path
to scalar-only queries. This is a deliberate compatibility hardening: a
mid-iteration driver error no longer exposes a partial scalar prefix, and
metric-construction errors are reported instead of panicking. It does not
change successful scalar metric names, types, labels, or values.

Negative observations are valid at the generic aggregation layer. Reference
time-age and XID-horizon collectors MUST clamp or filter impossible negative
values in SQL.

### 6.3 Missing columns

A configured Histogram column missing from the SQL result is a collection
error. A missing Histogram column MUST NOT silently produce an empty
distribution.

### 6.4 Empty populations

A label tuple with zero valid observations for a Histogram is not emitted.
Version 1 does not synthesize empty label tuples that are absent from SQL.
Reference SQL MAY generate explicit population rows when stable zero-valued
series are required, but NULL still contributes no observation.

### 6.5 Unsupported inputs

Version 1 does not accept:

- SQL arrays of bucket counts;
- one row per bucket boundary;
- weighted observations;
- cumulative counts supplied by SQL;
- delta windows;
- quantiles supplied by SQL.

These inputs require a separate, future input-layout contract.

## 7. Aggregation algorithm

### 7.1 Group identity

The aggregation key is the pair:

```text
(Histogram column identity, complete ordered label tuple)
```

The label tuple MUST use a collision-safe representation. A plain delimiter
join is not sufficient because label values may contain the delimiter.

### 7.2 Accumulator

Each logical Histogram accumulator contains:

```text
labels       ordered label values
count        uint64 observation count
sum          float64 observation sum
bucketCount  one uint64 count for each configured finite upper boundary
```

`bucketCount[i]` is cumulative and counts all observations `<= bucket[i]`.
An implementation MAY store disjoint bucket populations internally, but it
MUST convert them to cumulative values before exposition.

### 7.3 Observation assignment

For every finite observation `v`:

1. increment `count`;
2. add `v` to `sum`;
3. find the first configured boundary `b` for which `v <= b`;
4. increment that bucket and every larger cumulative bucket.

An implementation SHOULD use binary search plus either direct cumulative
updates or a final prefix sum. Exact-boundary observations belong to that
boundary's bucket.

### 7.4 Determinism

Emission order MUST be deterministic for tests and human-readable output:

1. configured finite boundaries in ascending order;
2. generated `+Inf` bucket;
3. count;
4. sum.

Logical Histogram groups SHOULD be emitted in deterministic metric-name and
label-tuple order where practical.

## 8. Prometheus exposition

### 8.1 Gauge bucket layout

Version 1 exposes each component with `prometheus.GaugeValue`:

```text
# HELP pg_session_query_age_seconds_bucket Current active query age snapshot in seconds (cumulative bucket)
# TYPE pg_session_query_age_seconds_bucket gauge
pg_session_query_age_seconds_bucket{datname="app",state="active",le="0.1"} 7
pg_session_query_age_seconds_bucket{datname="app",state="active",le="1"} 11
pg_session_query_age_seconds_bucket{datname="app",state="active",le="+Inf"} 12

# HELP pg_session_query_age_seconds_count Current active query age snapshot in seconds (observation count)
# TYPE pg_session_query_age_seconds_count gauge
pg_session_query_age_seconds_count{datname="app",state="active"} 12

# HELP pg_session_query_age_seconds_sum Current active query age snapshot in seconds (observation sum)
# TYPE pg_session_query_age_seconds_sum gauge
pg_session_query_age_seconds_sum{datname="app",state="active"} 8.73
```

The `+Inf` bucket value MUST equal `_count`.

The exporter MUST create separate descriptors for bucket, count, and sum.
The bucket descriptor has the query labels followed by the generated `le`
label. Constant labels apply to all three derived families.

Finite `le` label values MUST be formatted with:

```go
strconv.FormatFloat(boundary, 'g', -1, 64)
```

The generated infinite boundary MUST use the literal `+Inf`. This formatting is
part of the public series identity and MUST be covered by golden tests.

Prometheus quantile interpolation has a separate convention from bucket
storage. When the first finite upper boundary is positive and a quantile falls
in that first bucket, `histogram_quantile()` assumes `0` as the interpolation
lower bound. It does not treat the first bucket's storage interval as
`[0, b0]`, nor does it require an exported `-Inf` bucket. For a signed domain
whose first upper boundary is non-positive, Prometheus returns that upper
boundary when the quantile falls in the lowest bucket rather than extrapolating
toward negative infinity. Consumers MUST account for this lowest-bucket rule;
finite negative boundaries and an explicit zero boundary are therefore
important.

### 8.2 Deliberate metadata limitation

Version 1 does not emit:

```text
# TYPE <base> histogram
```

and does not emit:

```text
# TYPE <base> gaugehistogram
```

The first would misrepresent snapshot buckets as counters. The second requires
an OpenMetrics-aware gatherer and protocol negotiation not provided by the
current `client_golang` collector path.

The Gauge bucket layout preserves correct temporal behavior and remains usable
with classic `histogram_quantile()` because Prometheus evaluates classic
bucket series by the `le` label after ingestion.

Prometheus lint tools may warn that Gauge families use the reserved Histogram
suffixes `_bucket`, `_count`, and `_sum`. Version 1 accepts this warning as the
cost of preserving familiar bucket queries without falsely declaring a
counter-like Histogram. It is not a scrape or PromQL error. Documentation and
CI MUST NOT claim that this layout passes Histogram-specific promlint rules.

### 8.3 Cardinality

For `B` configured finite boundaries and `L` populated label tuples, a logical
Histogram emits:

```text
(B + 3) * L
```

series: `B` finite buckets, one `+Inf` bucket, one count, and one sum.

Reference collectors MUST use low-cardinality grouping labels. They MUST NOT
use `pid`, raw query text, relation name, or other per-object identity labels.

### 8.4 Internal scrape metrics

The query result-size metric SHOULD count the actual number of emitted
Prometheus component metrics. With the Gauge bucket layout, each bucket, count,
and sum is already one `prometheus.Metric`, so no hidden expansion adjustment
is required.

## 9. Cache, reload, and concurrency semantics

Histogram collection reuses the existing query lifecycle:

- A real SQL execution creates a new empty aggregation map.
- Successful completion atomically replaces the cached result with the fully
  buffered scalar and Histogram component metrics.
- A failed execution leaves an empty result for that execution and exposes no
  partial new result. Atomic result publication is a new version 1 guarantee;
  existing scalar row processing does not already provide it.
- A TTL cache hit sends the cached component metrics unchanged.
- A cache hit MUST NOT observe the cached SQL rows again.
- Configuration reload discards prior collector instances and Histogram
  snapshots through the existing re-plan lifecycle.
- Existing collector locks protect Histogram state in the same way as scalar
  metric results.

No background sampler, ticker, persistent state, scrape deduplication, or
cross-scrape accumulation is permitted in version 1.

The existing collector updates its last-execution timestamp after a real
attempt, including a failed attempt. Version 1 retains that behavior: a failed
non-fatal Histogram query caches the empty result until its TTL expires. During
that interval, absence means "no successful snapshot is available", not
necessarily an empty PostgreSQL population. Operators MUST use the existing
query error self-metrics to distinguish collection failure from absence. A
future retry-on-error policy requires a separate lifecycle change.

## 10. Reference PostgreSQL collectors

Version 1.2 is accepted only when it supports the `pg_session` reference query
family and its four logical Histograms.

### 10.1 Session snapshot Histograms

The compact `pg_session` namespace makes the exported names suitable for
routine dashboard and alert use. Its execution controls are normative:

```yaml
pg_session:
  name: pg_session
  ttl: 10
  min_version: 100000
  tags: [cluster]
```

The `cluster` tag is required because `pg_stat_activity` is cluster-wide. It
prevents auto-discovered database connections from emitting duplicate series.
The minimum version is PostgreSQL 10 because the reference SQL uses
`backend_type`.

PostgreSQL 10+ reference SQL:

```sql
SELECT datname,
       state,
       CASE
         WHEN state = 'active' AND query_start IS NOT NULL
         THEN greatest(0, extract(epoch FROM statement_timestamp() - query_start))
       END AS query_age_seconds,
       CASE
         WHEN xact_start IS NOT NULL
         THEN greatest(0, extract(epoch FROM statement_timestamp() - xact_start))
       END AS xact_age_seconds,
       CASE
         WHEN backend_start IS NOT NULL
         THEN greatest(0, extract(epoch FROM statement_timestamp() - backend_start))
       END AS age_seconds,
       CASE
         WHEN backend_xid IS NOT NULL OR backend_xmin IS NOT NULL
         THEN greatest(
           0,
           coalesce(age(backend_xid), 0),
           coalesce(age(backend_xmin), 0)
         )
       END AS xid_age
FROM pg_stat_activity
WHERE pid <> pg_backend_pid()
  AND backend_type = 'client backend'
  AND datname IS NOT NULL;
```

The monitoring role MUST be able to see activity details for other sessions;
membership in `pg_monitor` (or equivalent `pg_read_all_stats` privileges) is
the reference deployment requirement. The SQL still guards nullable timestamp
and XID fields so masked or unavailable values cannot become false zero-valued
observations.

Required labels:

```text
datname, state
```

The reference collector MUST NOT label by `pid`, `usename`, `application_name`,
`query_id`, `wait_event`, or query text.

### 10.2 Shared time-age buckets

All three second-valued observations MUST use the same boundaries:

```yaml
- query_age_seconds:
    usage: HISTOGRAM
    bucket: [0.01, 0.03, 0.1, 0.3, 1, 3, 10, 30, 100, 300, 1000, 3000, 10000, 30000, 100000, 300000, 1000000, 3000000]
    description: Current active query age snapshot in seconds

- xact_age_seconds:
    usage: HISTOGRAM
    bucket: [0.01, 0.03, 0.1, 0.3, 1, 3, 10, 30, 100, 300, 1000, 3000, 10000, 30000, 100000, 300000, 1000000, 3000000]
    description: Current transaction age snapshot in seconds

- age_seconds:
    usage: HISTOGRAM
    bucket: [0.01, 0.03, 0.1, 0.3, 1, 3, 10, 30, 100, 300, 1000, 3000, 10000, 30000, 100000, 300000, 1000000, 3000000]
    description: Current client session age snapshot in seconds
```

The boundaries form a compact engineering logarithmic grid. Within each
decade, the finite upper bounds are `1` and `3` times a power of ten. Adjacent
ratios therefore alternate between `3` and approximately `3.33`, rather than
varying from `2` to `7` as they did in version 1.1. The grid spans `10 ms` to
`3,000,000 s` (about `34.7 d`), uses 18 finite boundaries, and emits 21 series
per populated label tuple including `+Inf`, count, and sum.

This grid is a **distribution resolution lattice**, not a list of operating
policies. Pigsty and PostgreSQL thresholds such as `100 ms`, `1 s`, `60 s`,
`600 s`, `1 h`, or `1 d` remain useful dashboard and alert context, but they do
not all become bucket boundaries. Values such as `100 ms`, `1 s`, `10 s`,
`30 s`, and `300 s` happen to align with the lattice; that coincidence is not
a rule for adding future boundaries.

The compact grid deliberately trades some short-range quantile precision for
stable visual proportions and bounded cardinality. These metrics describe
live snapshot ages rather than completed request latency SLOs, so this is the
appropriate default tradeoff. A deployment that requires an exact count above
a non-boundary policy threshold SHOULD use an existing maximum-age Gauge or a
dedicated threshold Gauge. It SHOULD NOT distort the shared Histogram lattice
with every local policy value. Dashboards SHOULD treat `le` as numeric and use
a logarithmic value axis for distribution plots.

Using identical boundaries for query, transaction, and session ages keeps
their dashboards and recording rules comparable.

These observations are **live ages**, sampled from sessions that still exist
at collection time. They are not completed-query, completed-transaction, or
completed-session latency distributions. Long-lived observations are more
likely to appear in repeated snapshots, so statistical interpretation is
length-biased. An instant quantile answers a question about the current live
population only.

The `query_age_seconds` population includes only `state='active'` rows.
`query_start` MUST NOT be interpreted as the current query start for other
states.

### 10.3 Backend XID-horizon buckets

The backend horizon observation MUST use these boundaries:

```yaml
- xid_age:
    usage: HISTOGRAM
    bucket: [1000000, 2000000, 5000000, 10000000, 20000000, 50000000, 100000000, 200000000, 500000000, 1000000000, 2000000000]
    description: Current backend oldest XID horizon age snapshot in transactions
```

For each backend, `xid_age` is the greater non-negative age of `backend_xid`
and `backend_xmin`. It is absent only when both fields are NULL. Including both
fields matters: an open transaction may own an old XID, while a backend may
hold an older snapshot horizon through `backend_xmin`.

The boundaries form a `1, 2, 5` engineering logarithmic grid from `1m` to
`2b`. Adjacent ratios alternate between `2` and `2.5`. Values at or below `1m`
share the first finite bucket; they are normal for this risk-oriented view.
The generated `+Inf` bucket isolates observations above `2b` as the final
emergency band. Eleven finite boundaries emit 14 series per populated label
tuple.

The regular grid naturally aligns with several familiar PostgreSQL and Pigsty
risk landmarks:

- `50m` is PostgreSQL's default `vacuum_freeze_min_age`, below which a routine
  vacuum normally avoids freezing otherwise-recent XIDs;
- `100m` is the default `AGE_THRESHOLD` of Pigsty's bundled `pg-vacuum` tool,
  which proactively selects aging databases and relations when the tool is
  invoked or scheduled. It is an operator-tool policy, not a PostgreSQL GUC or
  an automatic server-side trigger;
- `200m` is PostgreSQL's default `autovacuum_freeze_max_age`;
- `500m` is a logarithmic intermediate observation line;
- `1b` is Pigsty's configured `autovacuum_freeze_max_age` for its standard
  PostgreSQL profiles and is also a prominent critical dashboard boundary;
- `2b` begins the final emergency band before the signed `2^31` XID-age limit.

Other important values remain policy context rather than bucket boundaries:
`150m` is PostgreSQL's default `vacuum_freeze_table_age`; `1.6b` is PostgreSQL
14+'s default `vacuum_failsafe_age`; approximately `1.7b` is Pigsty's 80%
database-age critical line; and approximately `2.1b` is the PostgreSQL 14+
warning neighborhood. Keeping these closely spaced values out of the bucket
lattice prevents extremely narrow bands near the top of a logarithmic plot.

These PostgreSQL parameters and Pigsty tool policies are **relation or database
frozen-XID landmarks**. A backend crossing one of these Histogram boundaries
does not itself trigger `pg-vacuum`, autovacuum, aggressive vacuum, failsafe
mode, or shutdown. The mapping is intentional operational context: an old
`backend_xid` or `backend_xmin` can hold back cleanup horizons, and a severe
backend age can frustrate the vacuum work intended to keep relation ages safe.
Operators MUST evaluate the Histogram together with actual database/relation
ages, effective GUC values, scheduled maintenance tools, autovacuum activity,
replication slots, and prepared transactions. Customized GUCs and tool options
may move the true maintenance thresholds; the stable bucket layout is not a
substitute for reading those settings.

The existing `pg_activity` count and maximum-duration Gauges remain unchanged.
The `pg_session` Histogram collector is additive and does not replace existing
series in version 1.2.

## 11. PromQL contract

### 11.1 Instant quantile

Current p95 active query age by database:

```promql
histogram_quantile(
  0.95,
  sum by (datname, le) (
    pg_session_query_age_seconds_bucket
  )
)
```

### 11.2 Population above a threshold

Current active queries older than 100 seconds, an exact version 1.2 boundary:

```promql
sum by (datname) (
  pg_session_query_age_seconds_bucket{le="+Inf"}
)
-
sum by (datname) (
  pg_session_query_age_seconds_bucket{le="100"}
)
```

`60 s` is a policy threshold but not a version 1.2 boundary. The Histogram
cannot provide an exact `> 60 s` population without interpolation. Consumers
that require that exact threshold SHOULD use a maximum-age or dedicated
threshold Gauge rather than matching a nonexistent `le="60"` series.

Consumers MUST use the boundary string actually exposed by version 1
formatting for exact matchers, for example `1e+09` rather than `1000000000`.

### 11.3 Snapshot mean

Current mean live transaction age:

```promql
sum by (datname) (pg_session_xact_age_seconds_sum)
/
sum by (datname) (pg_session_xact_age_seconds_count)
```

Consumers SHOULD guard division by zero or absent populations.

### 11.4 Prohibited queries

The following forms are semantically invalid for version 1 Histogram series:

```promql
rate(pg_session_query_age_seconds_bucket[5m])
increase(pg_session_query_age_seconds_count[1h])
irate(pg_session_xid_age_bucket[5m])
```

Recording rules and dashboards MUST NOT treat version 1 bucket, count, or sum
families as counters.

## 12. Error handling and observability

Configuration errors fail load with the query branch and column name in the
message.

Collection errors follow the existing query error and fatality behavior. A
Histogram-specific error SHOULD include:

- query name;
- Histogram column name;
- label tuple when safely printable;
- failure category, such as missing column, cast failure, or non-finite value.

The implementation MUST NOT log raw query text or sensitive label values merely
to diagnose Histogram aggregation.

No new self-metrics are required for version 1. Existing query duration,
execution count, error count, cache hit count, and emitted metric count are
sufficient.

## 13. Implementation boundaries

The expected implementation surfaces are:

- `exporter/column.go`: enable `HISTOGRAM`; retain `Bucket []float64`;
- `exporter/config.go`: classify Histogram columns and validate buckets and
  derived names;
- `exporter/query.go`: include logical Histograms in query metric metadata;
- `exporter/collector.go`: aggregate observations and emit component Gauges;
- `exporter/utils.go`: add or use a Histogram conversion path that distinguishes
  NULL-without-default from invalid/non-finite input; the existing
  `castFloat64` returns `NaN` for both and is not sufficient by itself;
- `exporter/validate_labels.go`: reject runtime constant label `le` when any
  parsed query contains a Histogram, using the existing validation call sites
  during exporter creation and reload;
- `config/0000-doc.yml` and `README.md`: document the implemented syntax and
  snapshot PromQL contract;
- `config/0415-pg_session.yml`: provide the four reference session snapshot
  Histograms;
- focused unit and lifecycle tests.

Version 1 SHOULD NOT require changes to the `Server`, `Exporter`, registry,
HTTP handler, or exposition negotiation path.

## 14. Test and acceptance criteria

### 14.1 Configuration tests

Tests MUST cover:

- case-insensitive `HISTOGRAM` usage normalization;
- acceptance of Histogram-only queries;
- empty bucket rejection;
- unsorted and duplicate bucket rejection;
- `NaN`, `+Inf`, and `-Inf` bucket rejection;
- derived metric and `le` label collision rejection;
- rename, scale, and default parsing;
- default multiplied by scale, matching scalar behavior;
- explain/YAML metadata showing configured buckets.

### 14.2 Aggregation tests

Tests MUST cover:

- exact-boundary inclusion with `<=` semantics;
- values below the first and above the last finite bucket;
- generated `+Inf` bucket equal to count;
- correct count and sum;
- multiple label tuples;
- collision-safe label tuple identity;
- multiple Histogram columns in one row set;
- NULL ignored without default;
- NULL observed with default;
- scale applied before bucket assignment;
- invalid and non-finite observations failing the new snapshot;
- one invalid Histogram observation discarding the entire mixed scalar and
  Histogram query result;
- deterministic output order.

### 14.3 Exposition tests

Golden gather tests MUST verify:

- bucket, count, and sum are all `GAUGE` families;
- finite buckets and `+Inf` are present;
- finite `le` values use the specified Go `g` formatting and `+Inf` is literal;
- `le` is appended without disturbing configured label order;
- help strings and constant labels are correct;
- no base `HISTOGRAM` family is emitted;
- emitted metric count includes every component series.

### 14.4 Lifecycle tests

Tests MUST verify:

- a real execution rebuilds the snapshot from zero;
- a cache hit returns identical metrics without accumulation;
- a smaller later population decreases buckets correctly;
- a query failure never exposes a partial replacement;
- a failed execution caches absence until TTL expiry under the existing
  last-execution timestamp policy;
- reload discards old Histogram collectors and cached snapshots;
- concurrent collection remains race-free under the existing lock model.

### 14.5 Reference collector acceptance

The implementation is complete only when:

- the four `pg_session` Histograms gather from a supported PostgreSQL instance;
- the `pg_session` collector runs once with `tags: [cluster]` under
  auto-discovery
  and emits no duplicate series;
- the three second-valued Histograms expose the identical required boundaries;
- `xid_age` uses the greater available age from `backend_xid` and
  `backend_xmin` and is absent when both are NULL;
- direct `histogram_quantile()` returns plausible instant values;
- threshold subtraction equals the independently queried SQL count;
- existing Gauge and Counter metric names and values remain unchanged;
- existing tests and race tests pass.

## 15. Non-goals

Version 1 explicitly does not provide:

- separate GaugeHistogram and CounterHistogram configuration types;
- cross-scrape Histogram accumulation;
- completed-query latency distribution;
- exporter-managed event deduplication;
- SQL pre-aggregated bucket ingestion;
- formal OpenMetrics `gaugehistogram` metadata;
- Prometheus Native Histograms;
- OTLP Histogram or ExponentialHistogram export;
- exemplars;
- dynamic or automatically inferred bucket boundaries;
- exporter-computed quantiles;
- high-cardinality per-PID, per-query, or per-relation Histogram labels;
- PostgreSQL 9.x compatibility for the `pg_session` reference query.

## 16. Future evolution

Future work requires a concrete data source and a separate design amendment.

If PostgreSQL or an extension provides reliable pre-aggregated buckets, the
configuration MAY later gain an input layout such as `observations` versus
`buckets`. If monotonically cumulative data is actually available, the same
logical `HISTOGRAM` usage MAY later gain `mode: cumulative`; separate usages are
not required by this specification.

Formal OpenMetrics GaugeHistogram exposition MAY be added behind the existing
configuration through a gatherer or encoder adapter. Native and OTLP encodings
MUST explicitly represent temporality or flavor and MUST NOT infer it from
bucket values.

None of these future possibilities changes the version 1 snapshot contract.

The reviewer suggested placing the constant `le` check in CLI label parsing.

The review identified failure-cached absence as an operational tradeoff. The
final design retains existing lifecycle behavior for version 1 rather than
expanding scope into retry policy. It makes the behavior explicit and requires
operators to use existing query error metrics.

bucket names and PromQL. Neither is preferable for the stated version 1 scope.

### C.2 Resolution lattice versus policy overlay

Version 1.1 mixed two concerns in one boundary list:

- a stable measurement lattice for heatmaps, CDFs, and quantile interpolation;
- exact PostgreSQL, Pigsty, and human operating thresholds.

The resulting time-age ratios ranged from `2` to `7`; XID-age ratios ranged
from `1.05` to `10`. On a logarithmic value axis this creates very wide and
very narrow visual bands. Because a classic Histogram counts observations
rather than normalizing by numeric or logarithmic bucket width, the unequal
bands also make heatmap color intensity difficult to compare. Classic
`histogram_quantile()` uses linear interpolation inside each bucket, so the
same geometry causes uneven estimation resolution.

Version 1.2 makes the Histogram buckets a stable logarithmic resolution
lattice. Operational thresholds remain dashboard and alert policy. A policy
value becomes an additional bucket only after a concrete requirement for an
exact threshold population justifies the cardinality and visual cost.

### C.3 Normative grids and cardinality

The three second-valued Histograms share the 18-boundary `1, 3, 10` grid from
Section 10.2. Each populated label tuple emits 21 series. The XID Histogram
uses the 11-boundary `1, 2, 5` grid from Section 10.3 and emits 14 series per
populated tuple. For one hypothetical label tuple populated in all four
Histograms, the component-series count falls from version 1.1's `90` to `77`.

The time grid favors a compact, nearly equal Log10 visual spacing across more
than eight decades. The XID grid uses slightly finer spacing because the
`1, 2, 5` sequence naturally retains the most useful PostgreSQL and Pigsty
risk magnitudes. The lowest XID boundary is `1m`; lower ages are intentionally
coalesced into the normal first band.

Dashboard distribution panels SHOULD use a numeric logarithmic value axis.
Existing maximum-age Gauges, database-age metrics, or a future dedicated
threshold Gauge SHOULD carry exact policy alerts. A second threshold-shaped
Histogram is deliberately not introduced.

### C.4 Live acceptance evidence

The version 1.2 reference configuration was exercised on PostgreSQL 18.4 with
an isolated database and a 60-client, 6-thread pgbench workload limited to
approximately one scheduled transaction per second. Each transaction assigned
a real XID, ran one `pg_sleep` selected from `0.3`, `3`, `10`, and `30` seconds,
then held the transaction open with a client-side sleep from the same set. The
60-second run completed 64 transactions with zero failures.

The committed workload is `docs/design/histogram-pgbench.sql`. The exact
invocation was:

```bash
pgbench -h 127.0.0.1 -p 5432 -U postgres \
  -d pg_exporter_hist_v12 -n -c 60 -j 6 -R 1 -T 60 -P 10 -r \
  --random-seed=42 -f docs/design/histogram-pgbench.sql
```

One uncached exporter snapshot contained 55 pgbench sessions grouped as 33
idle, 14 active, and 8 idle in transaction. Its non-cumulative active-query
populations were `1`, `3`, `2`, and `8` in the `(0.3,1]`, `(1,3]`, `(3,10]`,
and `(10,30]` second bands. The idle-in-transaction xact-age populations were
`1`, `2`, and `5` in `(1,3]`, `(3,10]`, and `(10,30]`. All 22 backends holding
an XID were in the first `xid_age <= 1m` band, as expected for a safe short
test.

The live exposition verified all 12 derived families as Gauges, complete
finite boundary sets, monotonically cumulative buckets, `+Inf == count`, and
the required NULL population semantics. Query age appeared only for active
sessions; xact and XID age appeared only for active or idle-in-transaction
sessions; ordinary idle sessions emitted session age without a false zero XID
observation. After pgbench disconnected, every series for the isolated test
database disappeared on the next uncached scrape, confirming replacement
snapshot behavior rather than cross-scrape accumulation.

The test did not attempt to manufacture million- or billion-transaction XID
ages. It validates the XID family structure and first-band behavior, not every
upper risk band.
